API 2.3

This documentation is for Version 2.3 of our API. Version 2 is no longer supported and will soon be removed. See documentation for Version 3 of our API.

What kind of really useful stuff can I build?

You can access and work with most of our features allowing you to create solutions. We’ve listed some examples below.

Collect leads in web forms. Perfect for Lead Gen, you can drive traffic to a landing page and use a web form to collect and send the prospect’s details directly to OnePageCRM. Connect to our cloud app. The beauty of the cloud is being able to connect with other useful business apps to offer our users a seamless and hassle-free experience.
Extract specific contacts. Filter your contacts by tag to only get what you need from the OnePage database. Set or save details. Without accessing your account you can use the API to set next actions, add deals or notes to contacts.

Request message format

  • Data requests should be sent as a typical HTTP request (i.e. in query string for GET and DELETE requests and as a body in format param1=value1&param2=value2... for PUT and POST requests)
  • Character data must be encoded in Unicode (UTF-8). Any invalid data will be ignored.
  • Successful update operations require every element describing the given resource to be provided. Update operations will fail if all required elements are not provided, resulting in corresponding resource property being lost.
  • There are some restrictions regarding specific data types, these are outlined in the table below. (more specific information will follow in later sections).
Data type Notes
Time Must be sent as unix time (in seconds) (http://en.wikipedia.org/wiki/Unix_time)
Date Must be sent in in dd.mm.yyyy (e.g. 16.06.2011) format
Country Must be sent as ISO-3166 2-letter country code (ISO 3166-1-alpha-2 code) except Greece, where EL must be sent
Boolean Must be sent 0 (false) or 1 (true)
String Leading and trailing spaces will be ignored and removed from the value

Request authorization

In order to authorize request, client should send following HTTP headers (except login method).

Header name Content Description
X-OnePageCRM-UID Unique user ID inside system (technically, 192-bit number written in hex)
X-OnePageCRM-TS Current timestamp in unix time
X-OnePageCRM-Auth HMAC-SHA256 request signature (signature data has been described below)

Calculating request signature value

The Request signature is a HMAC-SHA256 hash (written in hex using lowercase for a-f) of a string which consists of 4 (for GET and DELETE requests) or 5 (for PUT and POST requests) dot-separated elements, which are (in this exact order):

  1. X-OnePageCRM-UID header value,
  2. X-OnePageCRM-TS header value,
  3. HTTP request method name, uppercase (GET, POST, PUT or DELETE),
  4. SHA-1 (written in hex using lowercase for a-f) hash of the full request URL,
  5. (only for PUT and POST requests) SHA-1 hash (written in hex using lowercase for a-f) has of the raw request body.

Request signature - sample

Let’s assume the following conditions:

  • Our UID is "4e0046526381906f7e000002"
  • The current timestamp written as unixtime is "1308640873"
  • We are updating a contact with ID "4d91d3ea6381904e44000026" and we're taking response in XML (so we are making PUT request to URL "https://app.onepagecrm.com/api/contacts/4d91d3ea6381904e44000026.xml"), raw request body is firstname=John&lastname=Doe.
  • Our API key is (Base64 encoded) "AJfSRLr7uhsa9lOIgKQ4Vu72zzg3QTE7pJL2iSeA6Mo="

In order to calculate the request signature, we have to first calculate the SHA-1 hash for the request URL (which is "f0f5acefc2d21a1d60cd8e7f52f157442419f5d7") and for the request body (which is "42aa106ea05cada5b20a49844c9ea5878432f4e0"). So, we have to sign following string: "4e0046526381906f7e000002.1308640873.PUT.f0f5acefc2d21a1d60cd8e7f52f157442419f5d7.42aa106ea05cada5b20a49844c9ea5878432f4e0". HMAC-SHA256 hash for such string and key is "54d40f0284a23552eae5c62f6e848e636980e88cd13e7e7b58cc43d0825cc307".

Now you can sign your request with these headers:

  • X-OnePageCRM-UID : 4e0046526381906f7e000002
  • X-OnePageCRM-TS : 1308640873
  • X-OnePageCRM-Auth : 54d40f0284a23552eae5c62f6e848e636980e88cd13e7e7b58cc43d0825cc307

How to obtain resources data

The OnePageCRM data model is based on 8 types of resources:

  • contact
  • contact action
  • predefined sales action
  • contact note
  • contact deal
  • custom filter
  • linked company
  • invoice

Besides of that, there is some number of additional, standard controllers (accessible through standard GET/POST HTTP requests); all of them are described further in this document. Operations on the resources must be made with the following HTTP requests (according to RESTful rules). Assuming, that we are working on contact resource and using XML format following methods should be used in order to make required actions (some additional, specific parameters will be described further in this document in subsections related to specific resources and actions):

  • getting list of all contacts: GET /api/contacts.xml
  • getting information about specific contact: GET /api/contacts/123456789abcdef.xml
  • creating new contact (contact data in request body): POST /api/contacts.xml
  • updating existing contact (contact data in request body): PUT /api/contacts/1234567890abcdef.xml
  • deleting contact: DELETE /api/contacts/1234567890abcdef.xml.

Response message format

Response data can be sent in 3 different formats: XML, JSON and YAML1 (all of these are returning the same set of data, but one of them has to be explicitly chosen during request, e.g. contacts list can be retrieved through URL /api/contacts.xyz, where xyz can be xml, json or yaml). Date, time and country codes are returned in the same way as they are required in request message format.

Every response document is a map/hash/associated table (name depends on the terminology of programming language used2) with the following elements:

Element name Content description
timestamp Current server time
status Status code of the operation result (0 means success, other codes mean fail); whole list of available status codes can be found in the end of this document.
message Human-readable message associated with status element
data Operation result; this element is optional and may not be present in the response. Every time, some operation result is described, it will be placed inside this element3
tags Optional element, provides information about current user tags state (i.e. names and usage counts). Every time, when it occurs, client view should be updated with this data. Element content will be described in one of the subsections below.
statuses Optional element provides information about current contact statuses state (i.e. status types and usage counts). Every time it occurs, client view should be updated with this data. Element content will be described in one of the subsections below.
sales Optional element, provides basic sales summary. Every time, when it occurs, client view should be updated with this data. Element content will be described in one of the subsections below.
notifications Optional element, visible when user will reach some account restrictions. It’s an array of strings with following possible values:
  • deals_limit_reached - user reached maximum allowed number of deals. Adding additional deals or contacts is not allowed and will be rejected by system
  • contacts_list_full - user reached maximum allowed number of contacts. Adding additional contacts or deals is not allowed and will be rejected by system
  • too_many_contacts - allowed number of contacts exceeded. Any edit action on contacts, notes, tags or actions is not allowed and will be rejected by the system. Only contact deletion is allowed
  1. 1) If format will not be specified or it will be invalid, server will return empty response with "406 Not Acceptable" HTTP status code
  2. 2) A term hash will be used in this document
  3. 3) Please note, that even if some specific API method is not returning any data, optional elements can be still present in output data

XML response document format

XML document specification does not provide strict data type definition, therefore the following rules apply for the response data:

  • character data is encoded in UTF-8 format
  • binary data is represented in Base64 format
  • document is root element name is named hash
  • empty (AKA null/nil) values are represented by ‘nil="true"‘ element attribute
  • integer values are marked by ‘type="integer"‘ element attribute
  • float values are marked by ‘type="float"‘ element attribute
  • array are marked by ‘type="array"‘ element attribute; typically such element is a plural form of noun which describes stored data and this element nodes name is a singular form of such noun

Additional sales element

This element provides short sales summary. It contains information which can be found in the footer in the web version of the application. It contains following data:

Name Type Description
target float Current sales target amount.
period string Current sales target period. May be month or quarter.
won float Total won deals amount for present period.
pending float Total pending deals amount.

Additional tags element

This element provides information about current user tags. It consists of three elements:

Name Type Description
system array So called system tags, i.e. contact tags which cannot be edited by user. All system tags used are:
  • recently_viewed - contacts, which have been recently viewed (in web application such operation is called “expanding”)
  • recently_edited - contacts, for which some property has been changed recently (i.e. status changed, tags added etc.)
  • recently_imported - contacts, which have been imported in last import operation
  • pending_deals - all contacts, which have some pending deal
  • waiting - number of contacts marked as waiting
  • starred - number of starred contacts
custom array So called custom tags, i.e. contact tags which can be edited by user
team array contains pendings, custom, waiting and imported elements for whole team
Every element of such array consists 2 elements:
Name Type Description
name string Tag name
count integer Usage count

Companies

Getting list of companies

URL: /api/companies.format

Method: GET

Companies are the way of linking a number of contacts together. We are distinguish two types of companies:

  • unlinked - which simply represent contacts, which are not linked with other contacts (i.e. company related to this contact has one staff member). Unlinked company ID is the same as the contact one. Only contacts with non-empty company name are assigned to unlinked companies.
  • linked - this type of company contains a bigger number of assigned contacts and has its own unique ID; contact which is assigned to linked company is called linked contact.

Linked companies are created automatically using company name, i.e. when you’ll create contact with company name which already exists in database, such contacts will be linked

Input parameters:
Name Type Description
linked boolean Information if linked (true) or (unlinked) companies should be shown, defaults to true.
page_number integer Listing page number (first page = 1), defaults to 1
page_size integer Listing page size (number from 1 to 100, defaults to 50)
Output data:
array of elements describing single company, where single element has following format:
Name Type Description
id string Company ID
linked boolean Information if company is linked one
name string Company name
contacts_count integer Number of contacts assigned to this company
phone string Company phone number
address string Address
city string City part of the address
state string State part of the address
zip string Zip code part of the address
country string Country code part of the address
Sample JSON server response for getting list of companies.

Sales method

Sales summary

URL: /api/sales/index.format

Method: GET

Input parameters:

This method doesn't require any inputs parameters.

Output data:
Name Type Description
breakdown hash Current sales breakdown data. It contains 3 elements with keys: 30days, 60days and 90days containg data respectively for 30, 60 and 90 days. Every element contains two keys: won and lost, which contain total deal amount of given type from given period
closed array List of last 10 closed deals. See section describing contact notes in order to get element content description.
pending array All pending deals. See section describing contact notes in order to get element content description.

hitting_history array Four-element array where every one contains sales hitting history for last 4 sales periods (quarters or months). First element contains data from the oldest period, and the last one contains data from the current one. Every element contains two elements: target which is historical sales target for given period and actual which is sales target worked out in given period (calculated as a sum of won deals amount).
